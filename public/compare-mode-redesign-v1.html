§ COMPARE MODE REDESIGN · MOCK V1

Compare mode — three states.

The current compare mode uses two separate chrome layers: a dark announcement banner above the grid and a floating pill below it. Both are structural problems — neither is anchored to the app's ruled system, and the numbered slots in the pill look like mobile pagination in a desktop UI.

The redesign collapses everything into a single slim subrow beneath the toolbar. No dark banner. No floating pill. The toolbar's COMPARE button gains a count when items are selected, and the COMPARE action lives in the subrow where you can see your selection state. Selected tiles show a border, no fill tint.

Selection mode: The dark banner is gone. Instead a slim 36px subrow sits between the toolbar and the grid — same warm paper bg but slightly shifted to `var(--bg-2)` so it reads as a secondary layer. Left side shows the count or instruction. Right side has four slots that fill as recipes are selected, plus Clear, Exit, and the Compare → CTA. The CTA is disabled (opacity 40%) until at least 2 recipes are selected.

Selected tiles: `outline: 2px solid var(--fg)` — no tinted background fill. The cream tint in the current build was vestigial card-thinking. The editorial register uses line-weight not fills to communicate state. Hovering an unselected tile in compare mode shows a ghost checkmark at 30% opacity as a tap affordance hint.

COMPARE in toolbar: When selection mode is active, COMPARE stays in the toolbar but activates — it gets the fg color and underline treatment and shows a count in parens when anything's selected: `COMPARE (3)`. This means users can also trigger comparison from the toolbar control, not just from the subrow CTA. Both do the same thing.

Comparison chart: The chart structure was already nearly right — ruled rows, tabular numbers, mono labels. Two changes: (1) the black top border on photo thumbnails is removed — it was a CSS artifact, probably a `border-top` on the image container that shouldn't be there. (2) I added a subtle "lowest value" highlight on each row — the lowest number gets slightly stronger color (full `var(--fg)` instead of the lighter default). This is a small functional enhancement that makes the comparison actually useful: you can scan down a row and instantly see which recipe wins on each metric. It's optional — easy to drop if you prefer all values equal weight.

The ghost thumbnail (Almond Flour Lemon Bars) uses the same bottom-left text treatment as the recipe grid — consistent with the rest of the app.

What I didn't change: The overall chart structure, type sizes, column widths, row heights, the BACK TO RECIPES nav, and the NUTRITION COMPARISON page title. These are already correct and don't need touching.
