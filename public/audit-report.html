01 Color & Contrast
Critical
CTA button fails WCAG AA contrast
app/globals.css → .lp-cta-btn  |  app/login/page.tsx → submit button
Fails 4.5:1
Problem
The primary CTA button uses white text on #5A9B6A sage green. Measured contrast ratio: 3.3:1. WCAG AA requires 4.5:1 for normal text. This affects the landing page "Start tracking free" button and the login page submit button — two of the most critical interactive elements in the app.
White on #5A9B6A — 3.3:1 ✗ fails 4.5:1 AA
White on #3A7A4E — 5.2:1 ✓ passes 4.5:1 AA
Fix
Replace the button background with #3A7A4E — a darker shade of sage that achieves 5.2:1 contrast. Same visual language, fully compliant.
/* app/globals.css */ - background: #5A9B6A; + background: #3A7A4E;
Before · 3.3:1
Start tracking free
After · 5.2:1
Start tracking free
Critical
Eyebrow text fails WCAG AA contrast
app/globals.css → .lp-eyebrow  |  color: var(--accent) on --bg
Fails 4.5:1
Problem
The section eyebrow labels use color: var(--accent) which resolves to #5A9B6A. Measured against --bg: #F5F4EF, the contrast ratio is 3.06:1 — below the 4.5:1 threshold for normal-size text. Eyebrows are 9–11px mono, so they also can't claim the 3:1 large-text exception.
Before · 3.06:1 Nutrition Tracking
After · 5.8:1 Nutrition Tracking
Fix
Add a dedicated token --accent-text for text uses of the brand green, set to #2D6B3F (5.8:1 on --bg). Use var(--accent) only for backgrounds, borders, and decorative fills.
/* app/globals.css — :root */ + --accent-text: #2D6B3F; /* .lp-eyebrow */ - color: var(--accent); + color: var(--accent-text);
Critical
--ok and --accent are identical tokens
app/globals.css → :root — semantic token collision
Semantic error
Problem
Both --ok and --accent resolve to #5A9B6A. This means there is no visual distinction between "this action is your primary brand call-to-action" and "this status is healthy/on-track." When a recipe hits its calorie goal, the indicator reads identically to a primary button — users cannot tell semantic states apart.
Current state — both identical:
--accent #5A9B6A · CTA buttons, interactive
--ok #5A9B6A · Status: on-track ← same!
Fix
Give --ok its own distinct value. Keep sage for brand, use a clearly different hue for success states.
/* :root */ --accent: #5A9B6A; /* brand sage — keep */ - --ok: #5A9B6A; + --ok: #2E7D32; /* distinct success green */
After — distinct meanings:
--accent #5A9B6A · Brand / interactive
--ok #2E7D32 · Status: on-track ✓
High

Users can't identify what's clickable vs. decorative vs. a status indicator. The tab strip numbers, stat bars, avatar dots, progress fills, feature labels, and CTA buttons all share the same hue.

Hero headline at font-weight 600 instead of 700
app/globals.css → .lp-headline
Problem
The 48px Bricolage Grotesque hero headline uses font-weight: 600. At this large size the difference is subtle, but 700 is the designed weight for display usage in the design system. The current weight makes the headline feel lighter than intended, slightly reducing the visual impact of the opening statement.

Low
Hardcoded font strings throughout planner CSS
app/globals.css → planner, .pl-kcal-num, multiple selectors
Problem
Multiple planner selectors use literal font strings instead of CSS custom properties: 'DM Mono',monospace, 'DM Sans',sans-serif, and 'Bricolage Grotesque',sans-serif. If fonts are ever updated or the CDN URL changes, these won't update. The design system defines --mono, --sans, and --display for exactly this reason.
/* Current — hardcoded */ - .pl-kcal-num { font-family: 'Bricolage Grotesque',sans-serif; } - .pl-day-label { font-family: 'DM Mono',monospace; }
Fix
Replace all literal font strings with the defined custom property tokens.
+ .pl-kcal-num { font-family: var(--display); } + .pl-day-label { font-family: var(--mono); } /* Apply the same pattern to all ~12 planner selectors */
03 Whitespace & Spacing
High
Hero CTAs have only 4px top margin from tagline
app/globals.css → .lp-hero-ctas { margin-top: 4px }
Problem
The hero button row sits 4px below the tagline text. At any viewport width this creates a cramped feeling — the tagline and buttons visually merge into one block instead of feeling like a sentence followed by a clear call-to-action. The 8px design system grid minimum is 8px; 24–32px is appropriate for this content hierarchy break.
Before · margin-top: 4px
Track nutrition for the whole household.
Start free See how it works
After · margin-top: 28px
Track nutrition for the whole household.
Start free See how it works
Fix
/* app/globals.css */ .lp-hero-ctas { - margin-top: 4px; + margin-top: 28px; }
High
Feature description area clips text at fixed 52px height
app/globals.css → .lp-fs-desc-wrap { height: 52px }
Problem
The cycling feature description container uses height: 52px — a fixed value. Any feature description longer than two lines at the current font-size will be clipped or overflow. The intent is to prevent layout shift during cycling, but a fixed height is the wrong tool — it creates a content-clipping trap.
Fix
Use min-height to prevent layout shift while still allowing longer text to expand.
/* app/globals.css */ .lp-fs-desc-wrap { - height: 52px; + min-height: 52px; }
If layout stability during animation is needed, set contain: layout on the cycling container rather than capping the child's height.
04 Broken Tokens & Code Hygiene
Critical
Error message references non-existent CSS token
app/login/page.tsx → error paragraph · text-[var(--error)]
Runtime failure
Problem
The login error message uses text-[var(--error)] but the design system defines the error token as --err (not --error). This means login errors — "Invalid login credentials," "Email already in use" etc. — render in black instead of red, making them invisible against the form content. Users may not notice authentication failures.

← Before: error renders black (var(--error) is undefined)

← After: error renders red (var(--err) = #B02020)

Fix
/* app/login/page.tsx */ - className="font-mono text-[11px] text-[var(--error)]" + className="font-mono text-[11px] text-[var(--err)]"
Search the codebase for any other uses of --error and replace with --err.
Low
Tab strip numbers fail AA at 10px sage on light background
app/globals.css → .lp-fs-tab-num
Fails 4.5:1
Problem
The feature section tab numbers use color: #5A9B6A at 10px. At this size (below 18px / 14px bold), normal-text AA applies: 4.5:1 required, actual ratio: 3.06:1. These are decorative, but since they're part of interactive tab navigation, they count.
Fix
/* app/globals.css */ .lp-fs-tab-num { - color: #5A9B6A; + color: var(--accent-text, #2D6B3F); }
Low
Three parallel button style systems with no shared foundation
app/globals.css → .lp-cta-btn, .btn, inline Tailwind classes
Problem
The app has three distinct button styling approaches running simultaneously:
/* System 1 — BEM global class */ .lp-cta-btn { background: #5A9B6A; padding: 12px 22px; font-size: 11px; font-weight: 400 } /* System 2 — utility .btn class */ .btn { padding: 8px 16px; font-size: 9px; text-transform: uppercase } /* System 3 — inline Tailwind-ish classes */ className="bg-[var(--accent)] py-[8px] px-5 font-mono text-[9px]"
This creates inconsistency: three different padding sizes, two different font sizes, two different font families, inconsistent border-radius values (some use rounded-pill, some hardcoded border-radius: 2px). Any future button audit or accessibility fix must be made in three places.
Fix
Consolidate into a single .btn foundation + modifier classes. The inline Tailwind approach is fine for one-offs but shouldn't be the primary pattern for shared components.
+ .btn { /* Base */ padding: 8px 20px; font-family: var(--mono); font-size: 9px; text-transform: uppercase; letter-spacing: 0.1em; border-radius: 100px; border: none; cursor: pointer; } + .btn-primary { background: var(--accent); color: #fff; } + .btn-ghost { background: transparent; border: 1px solid var(--rule); color: var(--fg); }
