Design notes
Welcome
Animation rationale
  • Full stagger warranted — this screen is seen exactly once, Emil's framework says "rare/first-time events can add delight"
  • 5 layers stagger at 70ms intervals: bar → eyebrow → headline → body → CTA
  • Entry uses strong ease-out cubic-bezier(0.23, 1, 0.32, 1) — feels immediate
UX decisions
  • Single CTA — no decision paralysis on entry
  • "Skip" is below the fold visually but not hidden — power users who'll explore on their own should find it
  • No illustration or hero image — lean on typography and brand. The serif headline is the hero.
  • The 28px accent bar anchors the brand before the text appears — creates anticipation
Profile
Theme picker
  • 22px dots with 3px outline-offset ring on selection — concentric radius preserved (dot is circle, ring is circle)
  • Live CSS variable update on click — makes the choice feel personal immediately, not hypothetical
  • Warm vs cool themes naturally separate in the grid (warm top, cool bottom) — matches lib/themes.ts order
  • Hover scale: 1.18 — slightly playful, within tasteful range
UX decisions
  • Name pre-filled from signup — confirms rather than asks, reduces friction
  • Tip box explains what the theme does — avoids "what does this do?" confusion
  • No "skip" on this step — it's so quick there's no reason to skip
Household
UX decisions
  • "You" card is always shown with accent border — establishes who they are, not an option they can remove here
  • Added members animate in — fadeUp on entry so the list doesn't jar
  • Mock auto-assigns a contrasting theme to each added person — shows the visual distinctiveness
  • "Just me for now" skip is prominent — household is optional, don't block progress
  • Copy: "Recipes and ingredients are shared — goals and plans are per-person" — this answers the #1 question new users have about the data model
Future consideration
  • Could let the user set the added person's name + theme here inline instead of after onboarding
  • Invite link flow (for remote household members) should live in Settings, not onboarding
Nutrition Goals
This is the most important step
  • The entire app is less useful without goals set — progress bars and dashboard are empty shells without targets
  • Presets lower the cognitive load of "what should my protein goal be?" — most users don't know
  • "Active" is pre-selected: slightly aspirational, but realistic. Anchors the default higher so users feel they have room to customize down
  • Calorie number in DM Serif Display — visually weighted, feels like a real number not a label
Goal card animation
  • Cards use scale(0.99) on press — subtle tactility
  • Icon bg transitions: empty → filled accent on selection. The icon itself doesn't change color (emoji), which actually works — the filled bg provides enough contrast signal
  • "Custom → In Settings" defers complexity without blocking progress
First Recipe
The "magic moment"
  • URL import is the killer feature reveal — paste a URL, get structured nutritional data back. This is what makes the app worth using.
  • Three-state import flow: idle → loading (spinner) → success with recipe name + count — each state is meaningful, not just a spinner
  • Loading state shows "Extracting ingredients…" not "Loading…" — specific language shows what's happening
  • Success state shows the recipe name — concrete, confirms it worked
UX decisions
  • .md file import removed from this step — too confusing for first-time users. It lives in the full recipe create flow.
  • Copy now sets expectations about pantry matching — "flagged ingredients" language prevents surprise when unmatched items appear
  • "I'll add recipes later" skip is clearly visible — don't block progress
  • Consider: auto-populate the URL field with a sample recipe so users see the magic even without their own link ready
Completion
Stagger animation
  • Checkmarks cascade in at 120ms intervals — creates a sense of accomplishment, like things being "checked off"
  • The last item (create meal plan) uses a grey/pending circle — signals "here's what to do next" without making them feel incomplete
  • No confetti — too casual for this app's editorial aesthetic
  • Each check item fades up from 6px below — subtle, not showy
UX decisions
  • "You're all set." is the entire heading — short, confident, final
  • Summary shows what was actually configured (name, theme, goal preset, recipe name) — concrete reinforcement
  • Single CTA — no secondary actions to dilute the moment
  • After clicking "Open Good Measure": land on dashboard, which should show goals populated and a CTA to "Create this week's plan"
Step transitions (try the 3 variants above)
  • Slide: translateX(28px) + opacity, 320ms strong ease-out. Spatial — feels like moving through a sequence.
  • Fade: Opacity only, 280ms. Neutral — good if slides feel too "wizard-y"
  • Rise: translateY(16px) + opacity, 320ms. Upward movement suggests progression. Elegant for a refined aesthetic.
Implementation notes
  • Onboarding lives at /onboarding — separate route, not a modal overlay
  • State persists in server session — if user closes mid-flow, they resume where they left off
  • Re-visiting onboarding after completion → redirect to dashboard
  • onboardingCompleted: boolean on Household model — set on step 5 CTA click
  • Auth callback should check this flag → redirect to /onboarding if false
